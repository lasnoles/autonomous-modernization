# Environment — Podman topology + monitoring

> Built by the **devops** role during [INTEGRATE](../stages/INTEGRATE.md). Every
> environment is **ephemeral, reproducible, and rootless** — materialized from
> the graph's `ExternalSystem` dependency facts. There is always something at the
> other end of every wire, so **integration is never skipped**.

| | |
|---|---|
| **Engine** | rootless **Podman** · `podman-compose -f environment.yaml up` |
| **Network** | dedicated `billing-e2e-net` (one isolated network per run) |
| **Manifest** | [`../../artifacts/environment.yaml`](../../artifacts/environment.yaml) (rendered by devops) |
| **Readiness** | deterministic — **health check on every service**, all green |
| **Stage** | [INTEGRATE](../stages/INTEGRATE.md) · certified by integration-verifier |

```
billing-e2e-net (dedicated podman network)

```

## Services

### Service under test
- **billing-svc** `localhost/billing-svc:run-2026-06-28-001` — the migrated
  Java 21 / Spring Boot 3.x service. Port `8080`, health `/actuator/health`.
  Payments seam ([CU-021](../change-units/CU-021.md)) runs **shadow-only** with
  the feature flag **default-off** (`PAYMENTS_SEAM_ENABLED=false`); v1 stays the
  live path.

### Real dependents (official images, seeded with fixtures)
- **postgres:16** — seeded from `fixtures/billing.sql` so E2E scenarios replay
  against realistic data.
- **rabbitmq:3-management** — broker discovered as an `ExternalSystem`; official
  image, management UI on `15672`.

### Placeholder mock — the unknown payment provider
- **mock-payment-provider** `localhost/mock-payment-provider:run-2026-06-28-001`
  — a **PLACEHOLDER** mock built by the tester role to the placeholder
  `InterfaceContract`. The real provider's API is **unknown**, tracked as
  [**GAP-007**](../gaps/GAP-007.md). It returns plausible typed responses and is
  **loudly labeled** with its gap (`gap=GAP-007`, `placeholder=true`). This is how
  the payments integration path is exercised end-to-end *without* skipping it —
  it stands in for the unknown system and satisfies the *no dangling integration*
  exit criterion.

### Observability stack
- **otel-collector** — receives OTLP traces/metrics/logs from billing-svc.
- **prometheus** — **scrapes `billing-svc`** (`/actuator/prometheus`); the
  Target-Spec SLOs become alert rules and panels.
- **grafana** — `billing-e2e` dashboards: p95 latency, error rate, invoice
  throughput.
- **loki** — log aggregation.

So end-to-end behavior is **visible, not assumed** — the
[E2E report](../e2e/e2e-report.md) reads these signals to certify SLOs.
