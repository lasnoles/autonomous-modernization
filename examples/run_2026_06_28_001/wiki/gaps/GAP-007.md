## The unknown

`billing-svc` calls an **external payment provider** to charge invoices, but
neither the Target Spec nor the repo declares the provider's API. Walking the
resolution ladder:

1. **Human gate / prior decision** — none recorded.
2. **Intent / Target Spec** — provider not declared.
3. **Reality / code & graph** — call sites *use* a charge operation but the wire
   contract is not in-repo (the SDK is a binary dependency).
4. **System contracts / defaults** — none cover this external system.
5. **Reasoned inference** — the shape is *inferable from call sites* with
   evidence → infer it, record **confidence 0.30**.

The ladder yields no authoritative answer, and the unknown is an **external
interface**. Policy action: **create a typed placeholder + register this Gap, and
build the integration against the placeholder.** The integration is **not
skipped**.

## The typed placeholder

A first-class, typed artifact — not a `// TODO`:

```jsonc
{
  "gid": "billing-svc::InterfaceContract::PaymentProvider@external",
  "kind": "InterfaceContract",
  "placeholder": true,
  "protocol": "http",
  "shape": {
    "POST /charge": {
      "request":  { "amount": "decimal", "currency": "string", "token": "string" },
      "response": { "status": "enum[approved,declined]", "providerRef": "string" }
    }
  },
  "confidence": 0.30,
  "source": "inferred-from-callsites",
  "gap": "GAP-007"
}
```

The shape is **plausible and typed**, inferred from the call sites and naming.
It is **loudly marked** (`placeholder: true`, low confidence, this Gap) and
**isolated behind the seam** introduced by [CU-021](../change-units/CU-021.md)
so swapping in the real contract later is an O(1)-style change, not a rewrite.

### Assumption

> Provider exposes a **synchronous** `POST /charge` returning
> `approved | declined` plus a `providerRef`.

## Runtime backing

The placeholder is exercisable, not just declared. A Podman container
**`mock-payment-provider`** serves the `POST /charge` contract in the staged
environment, so the seam's `v2` path, the dependent environment, and the
[E2E suite](../e2e/e2e-report.md) all run against a live mock. See
[environment](../environment/environment.md) and
[flow-charge-invoice](../flows/flow-charge-invoice.md).

## Status: OPEN, non-blocking

`status: placeholdered` → **OPEN** and **`blocking: false`**. The run does **not**
stall: the integration is built, mocked, and tested against the placeholder, and
the run reached COMPLETE. The gap stays open until real provider information
supersedes the placeholder.

## How it raises risk

Per the policy, a Gap with low confidence raises the risk of any ChangeUnit that
depends on it (`HAS_GAP` edge). GAP-007 contributes **`raisesRiskBy 0.15`** to
[CU-021](../change-units/CU-021.md):

```
CU-021 base risk (blastRadius, coverage, criticality,
  equivalenceConfidence 0.65 golden-vs-placeholder, crossRepo)  = 0.37
GAP-007 gapRisk                                                  = +0.15
                                                          score  = 0.52  → pause band
```

So the gap is precisely what keeps CU-021 in the human-gated band rather than
auto-applying — the gate "naturally pauses changes built on shaky assumptions."

## What would resolve it

> The **provider's OpenAPI spec**, **or** **one captured production
> request/response pair**.

Either is enough to replace the placeholder with the real `InterfaceContract`.
On resolution: the placeholder is superseded, this Gap is marked `resolved`, and
**all work built on it is re-validated** (CU-021's golden replay re-run against
the real contract, equivalenceConfidence revised upward, CU-021 risk recomputed).
