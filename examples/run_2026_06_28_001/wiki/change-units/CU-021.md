## Rationale

The payments integration is the highest-criticality, least-understood part of
`billing-svc` — and it talks to an external provider whose API is unknown
([GAP-007](../gaps/GAP-007.md)). The objective calls for extracting it behind a
strangler-fig seam so the v2 implementation can coexist with v1 and be cut over
reversibly. The dependency-decouple in [CU-002](CU-002.md) (wave 1) prepared the
module boundary; the characterization tests in CU-027 captured current behavior
as a golden baseline.

## Targets (graph GIDs)

- `billing-svc::Component::Payments`
- `billing-svc::Class::com.acme.billing.payments.PaymentService`
- `billing-svc::InterfaceContract::PaymentProvider@external` *(placeholder — see [GAP-007](../gaps/GAP-007.md))*

## The seam

```yaml
seam:
  id: SEAM-payments
  kind: interface-facade
  boundary: billing-svc::Component::Payments
  introduces: com.acme.billing.payments.PaymentGateway
  routing:
    mechanism: feature-flag
    key: payments.v2
    default: off        # v1 remains the live path until deliberately flipped
  cutover: incremental
  rollback: flip-flag    # O(1), never a code revert (IR §7 invariant)
```

A `PaymentGateway` interface-facade is introduced; the legacy implementation
(`v1`) and the new extracted implementation (`v2`) both sit behind it. Routing is
controlled by the `payments.v2` feature flag, **default-off**, so applying the CU
changes no runtime behavior until the flag is deliberately flipped. Cutover is
incremental; rollback is an **O(1) flag flip**, satisfying the IR invariant that
seam-extraction CUs must declare an O(1) rollback (never a code revert).

## Dependency on GAP-007

This CU **depends on [GAP-007](../gaps/GAP-007.md)** — the external payment
provider's API is unknown. Per the open-question-resolution policy, the
integration was **not skipped**: it was built against a **typed placeholder**
`InterfaceContract` for `POST /charge`
(`{request: amount, currency, token}` → `{response: status[approved|declined],
providerRef}`), confidence **0.30**, source `inferred-from-callsites`. The seam's
`v2` path is wired to this placeholder, which is runtime-backed by the
`mock-payment-provider` Podman container (see
[environment](../environment/environment.md)). Because the gap is `OPEN` and
non-blocking, it **raises this CU's risk by +0.15** (`raisesRiskBy`), pushing the
score to 0.52.

## Equivalence

- **Level:** **golden (replay)**
- **Tests:** CU-027 characterization/golden tests
- **Replay:** true — recorded v1 request/response pairs are replayed against the
  v2 path through the placeholder to confirm identical observable behavior.

Equivalence confidence is **0.65** because the golden replay validates against
the *placeholder* contract, not the real provider — the residual uncertainty is
exactly what GAP-007 tracks.

## PRESERVES rules

- `billing-svc::Capability::Payments` — charge semantics
  (`approved`/`declined` outcomes, idempotency, provider reference capture)
  preserved; verified by golden replay.
- `billing-svc::Capability::Invoicing` — invoice-to-charge linkage unchanged
  (default-off flag means v1 stays live).
